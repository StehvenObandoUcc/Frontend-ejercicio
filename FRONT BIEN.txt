# Taller — Patrón Decorator

## 🎯 ¿Qué pide el Taller?

El profesor pide construir un **sistema de torres defensivas** (estilo Plants vs Zombies / Clash Royale)
donde cada torre base puede mejorarse con módulos adicionales, y cada mejora es un **Decorator**.
El frontend debe ser una **interfaz web embebida en Java**, usando `com.sun.net.httpserver` o Spark Java.

---

## 🏗️ Lógica del Patrón Decorator (Core del taller)

El sistema funciona así:

- Tienes una **Torre Base** que dispara proyectiles simples
- Le agregas mejoras encima como Decorators: dispara más rápido, congela enemigos, tiene escudo, genera oro, etc.
- Cada mejora es un **Decorator** que envuelve la torre anterior
- Al final, apilando múltiples Decorators, obtienes una torre súper poderosa

---

## 🌐 Lo que debe tener el Frontend Web

### Tecnología requerida

- Interfaz web **embebida directamente en Java**
- Servidor HTTP simple: `com.sun.net.httpserver` **o** Spark Java
- El frontend consume el backend REST con header `X-API-KEY: decorator-secret-2026`

### Funcionalidades que debe mostrar la web

| Sección | Qué debe hacer |
|---|---|
| **Catálogo de decoradores** | Listar los 4 decoradores disponibles: `INSURANCE`, `GIFT_WRAP`, `PRIORITY_SUPPORT`, `EXPRESS_DELIVERY` con su costo |
| **Crear componente/torre** | Formulario con `nombre`, `descripción` y `precio base` |
| **Aplicar decorador** | Botón/selector para aplicar un decorador sobre una torre existente en tiempo real |
| **Eliminar decorador** | Opción para remover un decorador específico de una torre |
| **Ver desglose de precio** | Mostrar precio base + costos de cada decorator + precio final |
| **Simular oleada de enemigos** | Simulación visual/funcional del comportamiento de la torre decorada |
| **Simulación sin guardar** | Endpoint `/simulate` — probar combinaciones de decoradores sin persistir |

### Endpoints del backend que el frontend debe consumir

| # | Acción visible en la web | Endpoint |
|---|---|---|
| 1 | Ver decoradores disponibles | `GET /api/v1/decorators` |
| 2 | Crear torre base | `POST /api/v1/components` |
| 3 | Ver todas las torres | `GET /api/v1/components` |
| 4 | Ver detalle de una torre | `GET /api/v1/components/{id}` |
| 5 | Aplicar mejora a torre | `POST /api/v1/components/{id}/decorators` |
| 6 | Quitar mejora de torre | `DELETE /api/v1/components/{id}/decorators/{type}` |
| 7 | Eliminar torre | `DELETE /api/v1/components/{id}` |
| 8 | Simular composición | `POST /api/v1/components/simulate` |

## ✅ Checklist de entrega del Frontend

Lo mínimo que debe cumplir la interfaz web:

- [ ] Se ejecuta embebida en Java (no React, no framework externo)
- [ ] Muestra el listado de decoradores con nombre y costo
- [ ] Permite crear una torre base con nombre, descripción y precio
- [ ] Permite aplicar decoradores uno a uno sobre una torre
- [ ] Muestra el desglose de precios tras cada decorador aplicado
- [ ] Permite eliminar un decorador de una torre
- [ ] Simula la torre en acción (oleada de enemigos) con los decoradores activos
- [ ] Toda comunicación con el backend incluye el header `X-API-KEY`

## 🔑 Reglas de código

Según el blueprint, aplican también al frontend HTML/JS embebido:

- Todo el código debe estar en **inglés**: nombres de variables, clases, endpoints y mensajes de error
- No se permite lógica de negocio en los controladores
